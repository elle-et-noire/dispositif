### MathJaxで配信量を最小限に抑えるための4つの最適化手法

Next.jsのSSG（静的サイト生成）などでMathJax (CHTML) を使う場合、以下の最適化を行うことで、KaTeXに近い、あるいはそれ以上の軽量化を実現できます。

#### 1. クライアントサイドのJavaScriptを「完全にゼロ」にする（No-JS / ハイドレーション無効化）

サーバーサイド（ビルド時）で完全にHTML/CSSへのレンダリングが終わっているなら、**ブラウザ側にはMathJaxのJavaScriptを一切ダウンロードさせない**のが最強の最適化です。

- **Next.jsでの実装イメージ:** ビルド時に数式をCHTMLに変換し、出力されたHTMLと「数式用CSS」だけをページに埋め込みます。ブラウザ側でMathJaxを起動するスクリプト（`<script src=".../mathjax.js">`）は**記述しません**。

- **効果:** これによりクライアント側のJS配信量は **0 KB** になります。


#### 2. 必要最小限の「CSS」だけを抽出してインライン化する

MathJaxが生成するCHTMLを正しく表示するためにはCSSが必要ですが、MathJax全体の巨大なCSSファイルを丸ごと読み込むのは無駄です。 MathJaxには、サーバーサイドでレンダリングした際に「そのページ（またはその数式）で実際に使われたCSSスタイルだけを抽出する機能（`MathJax.chtmlStylesheet()`）」があります。これをNext.jsのビルド時に抽出し、HTMLの`<style>`タグにインラインで直接埋め込みます。

- **効果:** 無駄なCSSのダウンロードがなくなり、HTMLサイズを最小限に抑えられます。


#### 3. フォントの自動高精度調整（`matchFontHeight`）をオフにする

ブラウザ側で動くJSを排除する（上記1を行う）場合、あるいはJSを残す場合でも、MathJaxの「周囲のテキストと高さを自動で合わせる計算処理」をオフに設定しておきます。

JavaScript

```
MathJax = {
  chtml: {
    matchFontHeight: false // 周囲のフォントとの高さ合わせを無効化
  }
};
```

- **効果:** これをオフにすることで、ブラウザ側の処理（CPU負荷）が劇的に軽くなり、余計なフォントメタデータの読み込みを防げます（サイズはCSSの `font-size: 1em` などで綺麗に連動するため、実用上の美観はほぼ損なわれません）。


#### 4. フォントの事前読み込み（`preload`）またはローカルフォントの利用

Webフォントがダウンロードされるまで数式が文字化け（または一瞬消える）する現象（FOIT/FOUT）を防ぐため、HTMLの`<head>`内で主要なMathJaxフォント（通常はローマン体やイタリック体のコアファイル）を `preload` 指定します。

HTML

```
<link rel="preload" href="/fonts/mathjax/MathJax_Main-Regular.woff2" as="font" type="font/woff2" crossorigin>
```

また、利用者の端末に「STIX Fonts」や「Latin Modern」などの数式フォントがすでにインストールされている場合は、サーバーからダウンロードせずにそれを使う設定（デフォルトで有効）にしておくことで、フォントの転送量を完全にゼロにできます。
